# Isometric Action Readability Rework Design

## Scope and Invariants

The existing one-match tower-defense rules remain unchanged: a 9 x 14 logical grid, 150 starting gold, 50-gold towers, 10 gold per kill, 25 gold per wave clear, 20 core HP, the existing wave counts/speeds/health, the existing tower damage/rate/range, and victory after wave five. Enemies still advance straight down their spawn column toward the shared core. The rework changes rendering, picking, depth presentation, and action feedback only.

The viewport remains 540 x 960 portrait with `canvas_items` stretch and `keep` aspect. Every visual is generated by GDScript drawing or built-in controls; no external image, font, audio, or generated asset is introduced.

## Coordinate Boundaries

`GridBoard` is the sole projection service. Game entities store continuous logical coordinates in grid units. Combat range, target selection, enemy travel, and projectile travel are computed in logical space. The view asks `GridBoard` to project those coordinates for drawing and node placement.

The board uses 64 x 32 pixel 2:1 diamonds. For a logical point `(col, row)`:

```text
screen_x = (col - row) * 32
screen_y = (col + row) * 16
```

The exact inverse is used before `floor` for picking. Cell centers are represented by `(col + 0.5, row + 0.5)`. A tap is first converted from viewport coordinates into the transformed `World` local coordinates, then inverse-projected and floored into a logical cell. This keeps camera framing, world scale, and shake out of picking math.

The entire 9 x 14 board is scaled uniformly to fit the portrait width and centered below the HUD. The core remains the shared rule target and is rendered under the center of the board's final edge; no pathfinding or per-column core logic is added.

## Rendering and Depth

`Grid` draws the back-to-front diamond field with purple enemy rows, neutral transit rows, and green ally rows. Occupied build tiles darken without changing their zone identity.

Enemies, towers, and projectiles live below one `EntitySort` node with `y_sort_enabled`. Each entity node is positioned at its projected ground contact point and draws its body upward from that point. This makes a lower entity naturally cover the raised body of an entity behind it. Projectiles retain a small positive relative z so their bright tracer remains readable.

`Fx` is a high-z world overlay outside the Y-sort container. It draws HP bars, damage numbers, selection diamonds, the projected range preview, invalid-placement marks, kill fragments, and leak telegraphing. These overlays therefore never disappear behind entity bodies.

## Action Language

- **Aim:** the teal tower barrel rotates continuously toward the current target; an idle tower is darker and centered.
- **Fire:** a short barrel recoil, pale muzzle diamond, and bright teal tracer begin together.
- **Hit:** the enemy body flashes white, receives a small view-only knockback opposite the projectile, and emits an orange damage number.
- **Kill:** the enemy shrinks while a warm orange procedural fragment burst expands from its ground point; the existing approximately 60 ms hit-stop remains.
- **Leak:** the core flashes red, a red downward slash appears between the enemy and core, and the world receives the existing decaying shake. It does not use the orange kill burst.
- **Placement:** the tapped diamond flashes teal and a projected range outline appears on success; invalid or unaffordable taps flash red with an X.
- **Wave start:** a short centered `WAVE X` banner slides/fades independently of combat effects.

Core, tower, enemy, and projectile silhouettes are distinct: a stepped cyan core, squat green base with narrow rotating teal barrel, purple raised enemy block with a bright health notch, and small luminous tracer respectively. State is communicated by color and motion, not surface detail.

## Runtime Responsibilities

- `GameConfig` retains all balance constants and adds projection/framing/feedback tuning constants plus pixel-to-grid derived values.
- `GridBoard` owns projection, inverse projection, board bounds, buildability, occupancy, diamond drawing, and core anchoring.
- `Enemy`, `Tower`, and `Projectile` own logical movement/combat state and receive a `GridBoard` view reference only to update their screen presentation.
- `Main` wires containers, converts taps through `World.to_local`, preserves the framed world position during shake, and routes semantic events to HUD/core/FX.
- `Fx` owns high-z transient feedback and overlay rendering. `Hud` owns the screen-space wave banner. `Core` owns its red damage flash.

Signals carry logical coordinates where event location matters. View-only effects never alter `grid_position`, target range, wave accounting, economy, or terminal-state resolution.

## Verification and Delivery

Headless rule tests assert exact projection examples, round-trip picking, logical enemy/projectile movement, unchanged balance values, and scene depth configuration. Flow tests continue to prove both five-wave victory and core-damage defeat. A true-headless timed scene smoke checks parser and runtime errors. Godot's headless display driver supports only the dummy renderer and cannot provide Viewport pixels, so portrait framing and visual presence are verified separately by an unattended render-capable capture: Xvfb with `gl_compatibility` in Linux CI and the native OpenGL display driver locally.

GitHub Actions remains the authoritative Godot 4.5 Android builder. After a successful run, the signed debug APK artifact is downloaded, signature-verified, and committed to `apk/godottest1.apk`. The workflow ignores APK-only pushes and the Android export excludes the `apk` directory so a published APK is never embedded inside the next APK.
